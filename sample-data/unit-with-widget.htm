Widget Konzept

Mit dieser Unit soll das Konzept für Verona-Widgets erklärt werden. Sie verwendet den simple player 6.0 in einer modifizierten Version. Es handelt sich lediglich um eine Konzeptskizze, daher werden auch mögliche Alternativen genannt.

Was ist ein Verona Widget?

Es handelt sich um ein, von der eigentlichen Unit unabhängiges interaktives Element, das

  • Unit-übergreifend verwendet werden kann und seinen Zustand unit-übergreifend behalten kann.
  • möglichst genau wie die player im Booklet-Ladevorgang geladen werden kann.
  • als Overlay über der eigentlichen Aufgabe (und sogar dem Host?) erscheinen kann.

Grundsätzlich verwenden Widgets dieselbe Technologie wie Verona-PLayer an sich: ein iframe, das srcdoc Attribut und postMessages.

Alternativ könnten a priori verschiedene Darstellungsweisen - z. B. "modal", "overlay", "sidebar", "movable" etc. festgelegt werden. Beim Aufruf des Widgets wird eine davon gesetzt. Für den Widgetautor würde das weniger Implementationsaufwand bedeuten, aber auch sehr viel weniger Funktionalität. Drag & Drop würde dann ggf. im Host umgesetzt werden. Das Widget müsste dennoch seine Größe selbst an den Host melden. Es könnte dann nur rechteckig sein. Eine Kombination aus beiden Konzepten ist evenfalls denkbar.

Kommunikation

Die Grundidee dieses Konzeptes ist, dass widgets konzeptuell nichts anderes sind als weitere, parallele Verona-Player-Sessions, die lediglich anders dargestellt werden.

Dies hat den Vorteil, dass die bereits etablierten Verona-Befehle verwendet werden können. Der Startvorgang ist der gleiche, wie bei einem Verona-Player:

  1. Eine vopReadyNotification das Widget teilt mit, dass es fertig geladen ist.
  2. Ein vopStartCommand, des Hosts fordert das Widget auf, zu starten. Es wird allerdings keine PlayerConfig mitgegeben, sondern eine WidgetConfig.

Widgets können dann, genauso wie der player selbst UnitDaten und -States oder -Logs mit vopStateChangedNotification vermelden, Fehler mit vopRuntimeErrorNotification melden und sogar eine Navigation mit vopUnitNavigationRequestedNotification anstoßen.

In dem vorliegenden Beispiel wurde auch vopWindowFocusChangedNotification verwendet, um einen Fokuswechsel ins Widget mitzuteilen, auch wenn, dieses in der nächsten Verona-Version wahrscheinlich nicht mehr vorkommt.

Was das Wegschreiben von Unitdaten angeht, muss man sich fragen, ob widgets immer zusätzliche units-chunks zur jeweils aktuellen Unit produzieren oder zu einer eigenen "virtuellen" Unit.
Der erste Ansatz hätte den Vorteil einer stärkeren Verzahnung von Unit und Widget, würde jedoch zu problemen führen, wenn Unit und Widget nicht dasselbe Datenformat verwenden, so lange Verona chunks mit verschiedenen Datentypen innerhalb einer Unit zulässt.

(Mehrheitlich) dieselbe API für widgets und player zu verwenden, führt dazu, dass die Widgets tendenziell sehr mächtig sind. Allerdings könnte eine Host-Implementierung bestimmte Funktionen für Widgets auch unterdrücken (z. B. Navigation), ohne dass dies vor vorne herein in der API-definition der Fall sein müsste.

Die Alternative wäre eine ganz neue API, die konzeptuell mehr oder minder die player API repliziert und definiert, wie der Host mit den Widgets kommuniziert.

Verwaltung

Widgets können über bestimmte Verona-Befehle aufgerufen und geschlossen werden.

Widget öffnen

{ "type": "vopStartWidget", "sessionId": "#1", "name": "calculator" }

Die Eigenschaft name referiert welches Widget geöffnet werden soll.

Widget schließen

{ "type": "vopCloseWidget", "sessionId": "#1", "widgetId": "calculator#1" }

Widgets auflisten

Der Player kann eine Liste der laufenden Widgets vom Host erhalten:

{ "type": "vopGetWidgetsRequestNotification" "sessionId": "#1", }

Vom Host gibt es eine Antwort, z. B.:

{ sessionId: '#1', type: 'vopGetWidgetsNotification', running: [ { id: "widget#1", name: } ], available: [ { name: 'calculator', }, { name: 'avatar', } ] }

Sie enthält zwei Listen:

  1. running: Eine liste von aktiven Widgets mit ID und name.
  2. available: Eine liste von Widgets, die aktuell geöffnet werden dürfen.
    Was das konkret bedeutet, hängt von der implementation im Host ab. im Testcenter, wird das wohl bedeuten "Widgets, die für diese Unit zugelassen sind".
    Der Host ist selbst dafür zuständig, Widgets, die nicht mehr gezeigt werden sollen (z. B. beim Unitwechsel) zu schließen - aus Schnittstellen sicht kann es also durchaus sein, dass in running völlig andere widgets stehen als in available, wenn dies nicht beachtet wurde.
    Außerdem muss der Host darüber entscheiden, ob zwei widgets mit dem selben Namen gleichzeitig gezeigt werden können. In dieser Testumgebung hier geht das, im Testcenter wäre das wahrscheinlich nict so.

Übergreifende Daten

BookletState

Widgets können bestehen bleiben, während die Unit gewechselt wird. Ein Widget kann auch verschwinden und soll dann wieder im selben Zustand geöffnet werden können (Beispiel: Avatar-Farbe). Hierzu wird ein übergeordnentes objekt verwendet, der bookletState, das mit der vopStateChangedNotification mitgegeben wird. Dieser kann von playern, vom host und vom widget gesetzt werden. Das Datenformat ist eine einfache Collection von Strings.

{ "avatar-type": "man", "avatar-color": "red" "calculatur-mode": "scientific" }
Es handelt sich also nicht um Variablen im engeren Sinne. Warum? Weil sich diese Datenstruktur im Testcenter bereits befindet. Den BookletState des Testcenters auch als bookletState für Verona einzusetzen, anstatt eine neue Datenstruktur einzuführen würde folgende Funktionen ohne weitere Implementierung mit sich bringen:
  • BookletState wird im Teststate bereits automatisch persistiert.
  • BookletState kann über die Testtaker-Datei personengenau vorbelegt werden.
  • BookletState kann im Review Modus frei gewählt werden.
  • BookletState wird im Gruppen-Monitor angezeigt und es kann danach sortiert udn gefilter werden.
  • BookletState beeinflusst die Verzweigung.
  • BookletState kann im debugmodus des TC live beorbachtet werden.
Die Alternative wäre eine neue Datenstruktur zu erfinden, die über die Units hinweg gehalten wird und jeder neuen unit und jedem neuen widget mitgeteilt wird. Aus Verona-sicht spricht nichts dagegen, aber es würde im Testcenter eine deutlich größere

WidgetConfig

Zusätzlich gibt es eine widgetConfig. Dieses Object beschreibt die config eines konkreten Widgets. Die einzige Eigenschaft, die in der vorliegenden Variante darin enthalten ist, ist clipPath, aber es sind natürlich noch andere Denkbar (siehe Darstellung).

  • clipPath: string - enthält einen string für die CSS-Eigenschaft clip-path
{ "clipPath": "ellipse(130px 140px at 10% 20%)" }

Entscheidend ist, dass die WidgetConfig vom widget gesetzt wird! Der Host wird darüber informiert in der vopReadyNotification oder eine speziellen vopChangeWidgetConfig.

{ "type": "vopChangeWidgetConfig", "widgetId": "w1", "widgetConfig": { "clipPath": "ellipse(130px 140px at 10% 20%)" } }

Implementation im IQB Testcenter und Studio

Deklaration in der Unit.XML

Die Unit.xml ist ein austauschformat für (Verona-)Units innerhalb der IQB Anwendungen.
Die folgenden Überlegungen betreffen also nicht Verona.

Das ein bestimmtes Widget für eine bestimmte Unit benötigt wird, muss in der Unit-Definition stehen (typischerweise Datei mit der Endung .voud.json. Wie das in der implementierung ist, ist Sache des Players. Zusätzlich sollte in der Unit-Deklaration (unit.xml) eine Abhängigkeit auf die Widgetdatei(-en) deklariert werden, damit diese vorgeladen werden können. Die Datenstrukturen dafür sind bereits vorhanden:

<Dependencies> <File for="player">calculator.html</File> </Dependencies>

Diese Dateien werden allerdings nicht vorgeladen. Dieser Fall ist allerdings bereits mitgedacht, und der Test-Loader lässt sich leicht entsprechend erweitern.

Aktuell wird das Feld Dependencies nur im Fall von Paketen wie GeoGebra oder einem Itemplayer-Paket verwendet, deren inhalte jedoch dynamisch nachgeladen werden. Es können jedoch auch einzelne Dateien angegeben werden. Idealerweise wäre es das XML-Schema um ein weiteres attribut zu erweitern, dass das ladeverhalten bestimmt.

Es wäre dann aufgabe des Studios und des Verona-Editors, bei Auswahl eines Widgets einen entsprechenden Vermerk in die Unit-Deklaration abzulegen.

Extra

Zum Abschluss noch ein Gedankenexperiment: Wenn Widgets nichts anderes sind, als parallele Verona-sessions mit etwas anderen Eigenschaften, könnte ein Widget auch ein normaler Verona-Player sein, dem man seinen Inhalt als Unit mitgibt. Widgets ließen sich somit auch z. B. im Aspect-Editor bauen.

Der Normalfall sollte aber natürlich sein, dass ein Widget ein single-unit Verona-PLayer ist.
